# 1.README
## サービス概要
家事の忘れや不公平感を解消し、パートナー間のコミュニケーションを円滑にする家事サポートアプリ。   
家事の時間になると同居メンバー全員に通知が届き、通知画面から「やります」ボタンを押すだけで参加できる。      
押した全員がポイントを獲得し、ご褒美と交換できる“二人で育てる家事習慣化ツール”。

## このサービスを作りたい理由・背景
ゴミ出しや洗濯など、毎週決まった家事を忘れることがあった。    
また、「なんとなく半々のつもり」でも実際には負担が偏っており、相手に無自覚に負担をかけていた経験がある。  
家事は見えづらく、やった・やっていないの認識差が大きいため、   
* 不満の種になる
* 感謝が伝わらない
* どちらか一方が疲弊する

といった問題が起きやすい。  
そこで、その時にできる人が気軽に家事に参加でき、協力した分だけ公平に評価される仕組みかつ、    
目に見える形でお互いに「ありがとう」を贈り合える工夫を持ったサービスを作りたいと考えた。  

## ユーザー層について
* 同棲中のカップル
  * 家事分担が曖昧になりやすい
  * お互いの頑張りを可視化すると喧嘩が減る
  * 「ご褒美」や「ケーキポイント」を使った遊び心が相性が良い
* 新婚・夫婦
  * 家事の持続的な不公平感を解消したい需要が強い
  * リマインド通知により家事忘れを防ぎたい層が多い
  * 感謝の仕組みを設けることで、関係の満足度が上がりやすい
* 忙しい共働き家庭
  * 互いに「余裕がある人がやる」運用が多く、家事の見える化が必須
  * スマホ通知からワンタップで家事記録できる利便性が刺さる

## サービスの利用イメージ
1. 家事の時間になると、スマホに通知が届く
2. 通知（ロック画面）から「やります」ボタンを押すだけ
3. 家事を行った人数分だけ、個別にショートケーキが貯まる
4. 8個貯まると自動でホールケーキに変換される
5. たまったホールケーキを消費して、「自分が欲しいご褒美」を相手にお願いできる
6. ご褒美が実行されたら、自然と感謝が伝わる関係になる  
### 得られる価値
* 家事忘れがなくなる
* 家事の不公平感がなくなる
* “ありがとう”が増えて雰囲気が良くなる
* 家事がゲーム感覚で楽しく続く

## ユーザーの獲得について
* SNS（Instagram / TikTok）でのショート動画広告
* App Store / Google Play のカップルカテゴリでのSEO施策

## サービスの差別化ポイント・推しポイント
* 家事を事前に分担しない “当番制ではない”
  * 通知が来たとき、その時に余裕のある人が「やります」と参加
  * 現実的で柔軟な運用（共働き家庭に特に合う）
* 家事した人全員がポイント獲得（手伝いも評価）
  * 「二人でやったのに片方だけポイント」は不満を生む
  * 全員が受け取れるため公平性が高い
* ポイントの管理は “共同” ではなく “完全に個人”
  * 「どっちがどれだけ頑張ったか」が数字で見える
  * 無駄な疑念や不満が起きにくい
* ご褒美は“自分のケーキで相手にお願いする”
  * “自分が頑張ったから自分がご褒美をもらえる”構造
  * モチベーションが高い、継続性が高い
* 通知から記録できるストレスフリー操作
  * アプリを開く必要なし

## 機能候補
* MVP（最小限でリリースしたい機能）
  * 家事リマインド通知
  * 通知から「やります」ボタン（ロック画面対応）
  * 家事実施人数分のポイント付与（ショートケーキ）
  * ショートケーキ → ホールケーキへの自動変換
  * ポイント（ケーキ）の個別管理
  * ご褒美リストの作成（自分の欲しいご褒美を事前に全員で共有）
  * ご褒美交換機能（ホールケーキで相手にリクエスト）
  * 家事カテゴリごとのポイント設定（重労働は多くするなど）
  * カレンダー連携

* 本リリースまでに追加したい機能
  * グラフ表示（週・月の家事量の可視化）
  * ペア以外の家族やルームシェアにも対応（人数制限解除）
  * 家事履歴（誰が何をしたか）

  * PostgreSQL
* デプロイ先
  * Render
* 使用予定のライブラリ

  * Firebase Cloud Messaging（FCM）（通知系）
  * Sidekiq + Redis（バックグラウンドジョブ）
  * Tailwind CSS（UI/UX）
  * Axios（React）（API コミュニケーション）
  * Hotwire（Rails）（API コミュニケーション）

### テーブル詳細
#### Users（ユーザー情報）
- id : 主キー
- groups.id : 所属グループID
- name : ユーザー表示名（例：山田太郎）
- email : ログイン認証用メールアドレス / ユニーク制約（例：taro@example.com）
- password_hash : パスワードハッシュ
- cake_amount : ケーキの所持数
- avatar_url : プロフィールのアイコン
- created_at : 登録日時
- updated_at : 更新日時

#### Groups（家事共有グループ）
- id : 主キー
- name : グループ名（例：山田家）
- created_at : 作成日時
- updated_at : 更新日時

#### Chores（家事マスタ）
- id : 主キー
- groups.id : グループID
- title : 家事名（例：皿洗い）
- detail : 家事の詳細
- cake_reward : 家事の報酬として得られるケーキ数
- created_at : 作成日時
- updated_at : 更新日時

#### Chore_dates（家事の実施予定日）
- id : 主キー
- chores.id : 家事ID
- occurrence_date : 実施予定日時（任意で複数選択可）

#### Chore_history（家事実行履歴）
- id : 主キー
- users.id : 実行したユーザーID
- chores.id : 実行した家事ID
- cake_count : 家事の報酬として得たケーキ数
- created_at : 作成日時

#### Rewards（ごほうびマスタ）
- id : 主キー
- groups.id : グループID
- users.id : 作成したユーザーID
- title : ごほうび名（例：マッサージ）
- cake_cost : ごほうびを得るのに必要なケーキ数
- created_at : 作成日時
- updated_at : 更新日時

#### Notifications（通知マスタ）
- id : 主キー
- chores.id : 通知する家事ID
- message : 通知内容
- notify_at : 通知する日時
- created_at : 作成日時

#### notification_users（ユーザーと通知の中間テーブル）
- id : 主キー
- user_id : 通知を受け取るユーザーID（Users.id）
- notification_id : 通知ID（Notifications.id）
